System Design
Built in layers.
Stays clean.

Five fully separated layers, each with a single responsibility, communicating only through defined interfaces, testable in complete isolation.

Group 6 — Architecture Diagram
Presentation
AuthScreen · EventDetailsScreen · ViewModels · Presenters
Logic
AuthService · EventService · PermissionManager · Validators
Domain Objects
Event · User · EventRecord · Location · Role
Persistence
SQLite · Stubs · Mappers · Table Definitions
Application
CampusShenanigans · DatabaseSeeder · Dependency Containers

Data Flow — 5 Steps

1
User ActionPresentation layer captures input and delegates requests to the logic layer.
2
Business LogicAuthService and EventService process requests using validation rules and permissions.
3
Interface CommunicationLogic interacts with Persistence through interfaces (e.g., EventDataPersistence) to maintain decoupling.
4
Persistence ImplementationConcrete implementations (SQLite or Stubs) map database rows to Domain Objects.
5
UI UpdateProcessed results propagate back to the presentation layer, updating the UI state.

Why This Works

The architecture enforces strict separation of concerns. Business logic remains completely independent of UI and storage, while persistence is interchangeable through interfaces. This allowed the team to switch from stub data to a real SQLite database with zero changes to the logic and presentation layers.

Postmortem.

A candid look at architecture decisions, process strengths, and the lessons that shaped our final release.

8
Features

What Went Well
  • We constantly evaluated our code for SOLID violations and design smells. This proactive approach prevented "technical debt" from piling up, ensuring we didn't have to rewrite huge chunks of logic every time we added a feature.
  • We maintained 100% pass rates across Unit, Mockito, Integration, and E2E tests. This safety net allowed us to refactor with confidence, knowing that existing functionality remained unbroken.
What Was Challenging
  • Early on, a large Service.java class began handling too many responsibilities across multiple layers. It took significant effort to untangle this "God Object" and redistribute its logic without breaking the system.
  • In early iterations, we relied heavily on Google Docs for coordination rather than GitLab Issues. This led to gaps in our official project history and made it difficult to accurately estimate our team's velocity.
ITERATION 2 RETRO
Improvement Area · Identified After Iteration 2

Eliminating Redundant Effort Through Ownership

What We Found

Due to a lack of centralized task tracking, two members unknowingly spent nearly 20 hours building the exact same administrative permission logic in isolated branches. We were forced to discard an entire branch's work during the merge, effectively cutting our team's productive capacity in half for that iteration.

What Changed

We moved 100% of tasks into GitLab Issues with a "No Claim, No Code" rule. We also implemented Domain Separation: we split the remaining features (Attendance & Favorites) into distinct layers where one member owned the Database/Logic while the other owned the UI, treating each other's files as "locked" to prevent overlaps.

Evidence It Worked

Iteration 3 saw zero redundant work. Every GitLab Issue traced directly to a single Merge Request. Because there were no competing logic blocks, merge conflicts were almost entirely eliminated, and final integration took minutes instead of hours.

Net outcome: Transitioning to a structured, issue-driven workflow recovered our full development capacity and resulted in a conflict-free final release.

Architecture Lesson

Design for the current need but stay open for growth. Our 5-layer split meant that swapping temporary stub data for a real SQLite database required zero changes to our UI or Business logic layers.

Process Lesson

Keep feature branches short-lived and narrowly focused. If you see something else that needs fixing, open a separate issue. Mixing multiple fixes in one branch is a reliable recipe for merge conflicts and lost work.

Teamwork Lesson

Constant communication is your real-time project tracker. Sharing availability allowed us to redistribute tasks proactively before deadlines and ensured that no part of the codebase became a "blind spot" for the team.
